# Database Keys

A **Key** is a column (or multiple columns) used to uniquely identify a row or create relationships between tables.

---

# 1. Primary Key (PK)

### What is it?

A Primary Key uniquely identifies every row in a table.

Rules:

- Cannot be NULL
- Must be UNIQUE
- Only one Primary Key per table

### Example

```sql
CREATE TABLE users (
    id SERIAL PRIMARY KEY,
    name VARCHAR(100)
);
```

### Output

| id  | name  |
| --- | ----- |
| 1   | Rahim |
| 2   | Karim |

### Real-world Use

- User ID
- Product ID
- Order ID

---

# 2. Foreign Key (FK)

### What is it?

A Foreign Key connects one table to another by referencing the Primary Key of another table.

### Example

```sql
CREATE TABLE orders (
    id SERIAL PRIMARY KEY,
    user_id INT REFERENCES users(id),
    amount DECIMAL
);
```

### Relationship

```
users
-------
id (PK)

↓

orders
-------
user_id (FK)
```

### Output

| Order | User  |
| ----- | ----- |
| 101   | Rahim |
| 102   | Rahim |
| 103   | Karim |

### Real-world Use

- Orders → Users
- Posts → Users
- Comments → Posts
- Reviews → Products

---

# 3. Candidate Key

### What is it?

A Candidate Key is any column that **can uniquely identify** a row.

A table may have multiple Candidate Keys.

Example

| id  | email           | username |
| --- | --------------- | -------- |
| 1   | rahim@gmail.com | rahim123 |

Here,

- id
- email
- username

All are Candidate Keys.

### Real-world Use

User Login

---

# 4. Alternate Key

### What is it?

Candidate Keys that are **NOT chosen** as the Primary Key.

Example

```
Primary Key → id

Alternate Keys

email

username
```

### Real-world Use

Login using Email

Login using Username

---

# 5. Composite Key

### What is it?

A key made from **multiple columns**.

One column alone cannot uniquely identify a row.

Example

| student_id | course_id |
| ---------- | --------- |
| 1          | 101       |
| 1          | 102       |
| 2          | 101       |

Here,

(student_id, course_id)

uniquely identifies each row.

### Real-world Use

Student Enrollments

Shopping Cart

Attendance System

---

# 6. Composite Primary Key

### What is it?

A Primary Key created using multiple columns.

### Example

```sql
CREATE TABLE enrollments (

student_id INT,

course_id INT,

PRIMARY KEY(student_id,course_id)

);
```

### Real-world Use

- Student Courses
- User Roles
- Product Orders

---

# 7. Surrogate Key

### What is it?

An artificial key generated by the database.

Usually an auto-increment integer or UUID.

Example

```sql
id SERIAL PRIMARY KEY
```

or

```sql
id UUID PRIMARY KEY
```

### Real-world Use

Almost every modern application.

Examples

- User ID
- Product ID
- Order ID

---

# 8. Natural Key

### What is it?

A key that already exists in the real world.

Examples

- National ID
- Passport Number
- Email
- Phone Number

### Example

| email           |
| --------------- |
| rahim@gmail.com |

### Real-world Use

Government Systems

University Systems

Passport Database

# Relationships

Relationships connect tables.

Three main types.

- One-to-One
- One-to-Many
- Many-to-Many

---

# 1. One-to-One (1:1)

Each record is connected to exactly one record.

Example

User

↓

Profile

```
Rahim → Profile

Karim → Profile
```

### Example

users

| id  | name  |
| --- | ----- |
| 1   | Rahim |

profiles

| id  | user_id | bio       |
| --- | ------- | --------- |
| 1   | 1       | Developer |

### Real-world Use

- User Profile
- Passport Details
- Employee ID Card

---

# 2. One-to-Many (1:N)

One parent can have many children.

Most common relationship.

Example

```
Rahim

↓

Order1

Order2

Order3
```

### Tables

users

| id  | name  |
| --- | ----- |
| 1   | Rahim |

orders

| id  | user_id |
| --- | ------- |
| 101 | 1       |
| 102 | 1       |
| 103 | 1       |

### Real-world Use

- User → Orders
- Category → Products
- Author → Posts
- Company → Employees

---

# 3. Many-to-Many (M:N)

Many records can belong to many records.

Needs a Junction Table.

Example

Students

↓

Enrollments

↓

Courses

---

students

| id  | name  |
| --- | ----- |
| 1   | Rahim |
| 2   | Karim |

courses

| id  | course   |
| --- | -------- |
| 101 | Database |
| 102 | Java     |

enrollments

| student_id | course_id |
| ---------- | --------- |
| 1          | 101       |
| 1          | 102       |
| 2          | 101       |

### Real-world Use

- Students ↔ Courses
- Users ↔ Roles
- Products ↔ Tags
- Movies ↔ Actors

# Referential Integrity

### What is it?

Referential Integrity ensures that every Foreign Key value points to a valid Primary Key.

Without it,

```
orders

user_id = 100
```

But

```
users

id = 100

doesn't exist ❌
```

Database rejects this.

---

Example

```sql
INSERT INTO orders(user_id,amount)

VALUES(100,500);
```

### Output

```
ERROR:

Foreign key constraint failed.
```

Because User 100 doesn't exist.

### Real-world Benefit

- Prevents Invalid Data
- Keeps Relationships Correct
- Maintains Data Consistency

---

# Primary Key vs Foreign Key

| Primary Key               | Foreign Key               |
| ------------------------- | ------------------------- |
| Uniquely identifies a row | References another table  |
| Cannot be NULL            | Can be NULL (if allowed)  |
| One per table             | Multiple allowed          |
| Must be UNIQUE            | Can have duplicate values |

---

# Surrogate Key vs Natural Key

| Surrogate Key         | Natural Key          |
| --------------------- | -------------------- |
| Generated by database | Exists in real world |
| Stable                | May change           |
| Integer/UUID          | Email, Passport, NID |
